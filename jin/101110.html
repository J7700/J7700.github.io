一：JWT鉴权

1：身份认证

1.1什么是身份认证？

身份认证（Authorization）又称”身份验证”、”鉴权”，是指通过一定的手段，完成对用户身份的确认。
日常生活中的身份认证随处可见，例如：高铁的验票乘车，手机的密码或指纹解锁，支付宝或微信的支付密码等。
在Web开发中，也涉及到用户身份的认证，例如：各大网站的手机验证码登录、邮箱密码登录、二维码登录等。

1.2为什么需要身份认证（怎么证明我是我）

身份认证的目的，是为了确认当前所声称为某种身份的用户，确实是所声称的用户。

1.3不同开发模式下的身份认证

对于服务端渲染和前后端分离这两种开发模式来说，分别有着不同的身份认证方案：
服务端渲染推荐使用Session认证机制—-当前端请求后端接口不存在跨域问题的时候，推荐使用Session身份认证机制。
前后端分离推荐使用JWT认证机制—-当前端需要跨域请求后端接口的时候，推荐使用JWT认证机制

2：什么是JWT

2.1介绍

英文全称：JSON Web Token 是目前最流行的跨域认证解决方案。
用户的信息通过Token字符串的形式，保存在客户端浏览器中。
服务器通过还原Token字符串的形式来认证用户的身份。
由于 Cookie 默认不支持跨域访问，所以，当涉及到前端跨域请求后端接口的时候，需要做很多额外的配置，才能实现跨域 Session 认证。

2.2原理


用户的信息通过 Token 字符串的形式，保存在客户端浏览器中。服务器通过还原 Token 字符串的形式来认证用户的身份

3：JWT组成部分

分别是由3部分组成 Header（头部）、Payload（有效荷载）、Signature（签名）。
三者之间使用英文的“.”分隔，
其中头部和荷载都是以JS0N格式存放数据，只是进行了编码。

第一部分：header

每个JWT都会带有头部信息，这里主要声明使用的算法。
声明算法的字段名为alg,
同时还有一个typ的字段，默认JWT即可。
以下示例中算法为HS256:
{“alg””HS256”,”typ”:”JWT”}
因为JWT是字符串，所以我们还需要对以上内容进行Base64编码，编码后字符串如下
eyJhbGcioiJIUzI1NiIsInR5cCI6IkpXVCJ9

:::info
第二部分：payload
:::
载荷即消息体，这里会存放实际的内容，也就是Token的数据声明，
例如用户的id和name,默认情况下也会携带令牌的签发时间iat,还可以设置过期时间
{
“sub”:”1234567890”,
“name”:”CodeJin”,
“iat”:1516239022
}
同样进行Base64编码后，字符串如下：eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6 IkpvaG.4gRG91IiwiaWF0IjoXNTE2MjM5MDIyfQ

第二部分：Signature

签名是对头部和载荷内容进行签名，一般情况，设置一个secretKey,对前两个的结果进行HMACSHA25算法
公式如下
Signature HMACSHA256(base64Url(header)+.+base64Url(payload),secretKey)

Payload 部分才是真正的用户信息，它是用户信息经过加密之后生成的字符串。
Header 和 Signature 是安全性相关的部分，只是为了保证 Token 的安全性。

4：客户端如何使用JWT

当客户端发起登录请求时，服务端会将登录信息转换为token字符串
客户端收到服务器返回的JWT之后，通常会将它储存在localstorage或sessionstorage中
此后，客户端每次与服务器通信，都要带上这个JWT的字符串，从而进行身份认证。
推荐的做法是把JWT放在HTTP请求头的Authorization字段中
